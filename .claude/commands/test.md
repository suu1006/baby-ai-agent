# 테스트 에이전트

`docs/spec-*.md` 명세를 기반으로 테스트 코드를 생성합니다.

## 실행 절차

### 1단계: 명세 파일 확인

`docs/` 폴더에서 가장 최근 `spec-*.md` 파일을 찾아 읽으세요.

명세 파일이 없으면: "먼저 `/spec`을 실행해 기능 명세를 작성하세요."라고 안내하고 종료하세요.

### 2단계: 테스트 대상 파악

명세를 분석해 테스트할 항목을 분류하세요:

- **단위 테스트**: 함수·유틸리티의 입출력 검증
- **컴포넌트 테스트**: 렌더링, 사용자 인터랙션, 상태 변화
- **통합 테스트**: API 호출, Supabase 연동 등

### 3단계: 테스트 코드 생성

`__tests__/{기능명}.test.tsx` 파일을 생성하세요.

**사용 라이브러리:**
- `jest-expo` (프리셋)
- `@testing-library/react-native` (컴포넌트 테스트)

**테스트 구조 원칙:**
- `describe` 블록으로 시나리오 그룹화
- `it` 설명은 "~하면 ~해야 한다" 형식으로 작성
- 외부 의존성(API, Supabase)은 `jest.mock()`으로 처리
- 정상 흐름 → 예외 흐름 순서로 작성

**기본 템플릿:**

```typescript
import { render, fireEvent, waitFor } from '@testing-library/react-native';

describe('기능명', () => {
  describe('정상 흐름', () => {
    it('~하면 ~을 보여야 한다', async () => {
      // Arrange
      // Act
      // Assert
    });
  });

  describe('예외 흐름', () => {
    it('~일 때 ~을 보여야 한다', async () => {
      // Arrange
      // Act
      // Assert
    });
  });
});
```

### 4단계: 테스트 실행 및 보고

```bash
npm test -- --testPathPattern={기능명}
```

결과를 사용자에게 보고하세요:

---
**[테스트 생성 완료]**

- 파일: `__tests__/{기능명}.test.tsx`
- 테스트 항목: N개
- 실행 결과: 통과 N / 실패 N / 스킵 N

실패한 테스트가 있다면 원인을 분석해 사용자에게 설명하세요.
통과했다면: "이제 기능을 구현하고, 완료 후 `/review` → `/push`로 진행하세요."

---

## 주의사항
- 명세에 없는 동작은 테스트하지 않습니다
- 구현 불가능한 테스트(하드웨어 의존, 실제 네트워크 등)는 mock으로 대체합니다
- 테스트 코드가 구현 코드보다 먼저 작성되어야 합니다 (TDD 원칙)
