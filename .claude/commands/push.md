# GitHub Push 서브에이전트

현재 작업 디렉토리의 변경사항을 검토하고, 사용자 승인을 받아 GitHub에 커밋·푸시합니다.

## 실행 절차

### 1단계: 변경사항 수집 및 1차 승인 요청

아래 명령어로 현재 변경사항을 수집하세요:

```bash
git status
git diff
git diff --cached
```

변경사항이 없으면 (git status가 clean) "커밋할 변경사항이 없습니다"라고 보고하고 즉시 종료하세요.

수집한 내용을 바탕으로 다음 형식으로 사용자에게 보고하세요

---
**[변경사항 보고]**

**수정된 파일:**
- 파일 목록과 각 파일의 주요 변경 내용 요약

**변경 내용 요약:**
- 변경의 목적과 영향 설명

이 변경사항을 커밋하시겠습니까? (승인/취소)

---

사용자가 취소하면 즉시 중단하세요. 승인하면 2단계로 진행하세요.

### 2단계: 커밋 메시지 작성 및 최종 승인 요청

**커밋 메시지 작성 규칙:**

타입 목록:
- `feat` : 새로운 기능 추가
- `fix` : 버그 수정
- `refactor` : 코드 리팩토링 (기능 변경 없음)
- `style` : 스타일·포맷 변경 (기능 변경 없음)
- `docs` : 문서 수정
- `chore` : 빌드·설정·의존성 변경
- `test` : 테스트 추가·수정

형식: `타입(scope?) : 한글로 간결하게 설명`

예시:
- `feat : 삭제정책 추가`
- `feat(ui) : 기록 통합 수정 및 모달 키보드 개선`
- `fix : 메세지 끊김현상 이슈 수정`

scope는 변경 범위가 명확할 때만 추가 (ui, api, auth 등).

변경사항을 분석해 적절한 커밋 메시지를 1~3개 제안하고 사용자에게 다음 형식으로 보고하세요:

---
**[커밋 메시지 제안]**

1. `feat : 예시 메시지`
2. `feat(ui) : 예시 메시지`
3. (직접 입력)

위 메시지 중 하나를 선택하시거나, 직접 입력해주세요. 취소하려면 '취소'를 입력하세요.

---

### 3단계: 커밋 및 푸시 실행

사용자가 최종 커밋 메시지를 확정하면:

1. 변경된 파일 전체를 스테이징:
```bash
git add -A
```

2. 커밋 실행 (node_modules는 제외, .gitignore 확인):
```bash
git commit -m "확정된 커밋 메시지"
```

3. 현재 브랜치 확인 후 원격 최신화 및 푸시:
```bash
git branch --show-current
git pull origin 현재브랜치명 --rebase
git push origin 현재브랜치명
```

4. 실제 커밋 해시 확인:
```bash
git log --oneline -1
```

5. 성공 시 결과를 사용자에게 보고:
---
**[완료]**
- 커밋: `커밋 해시 (git log --oneline -1 결과값)` — `커밋 메시지`
- 푸시: `origin/브랜치명`
- GitHub: https://github.com/suu1006/baby-ai-agent
---

## 주의사항
- 사용자가 어느 단계에서든 '취소'하면 즉시 중단
- `node_modules/`, `.env` 등 민감하거나 불필요한 파일은 스테이징하지 않음
- 커밋 전 반드시 최종 승인을 받을 것
- 강제 푸시(`--force`)는 절대 사용하지 않음
